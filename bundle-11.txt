# Inventory ERP

A lightweight, Fiori/Dynamics-style ERP inventory monitoring web app built with Next.js (App Router),
TypeScript, Tailwind CSS, AG Grid, and ECharts. It reads inventory data from an Excel workbook —
either a locally generated sample workbook (works out of the box) or a live SharePoint file via
Microsoft Graph API once you provide Azure AD credentials.

## What's included (Core ERP scope)

- Home executive dashboard with 12 KPI cards, health score gauge, category value chart, order-status chart, recent alerts
- Full Inventory grid (AG Grid): sorting, per-column filtering, resizing, pagination, CSV/Excel/PDF export, quick search, dropdown filters (category, supplier, classification, criticality, order status, brand)
- Automatic category detection from Item Code prefix (CO, PM, RM, FA, GB, FPB) and 6 category pages with their own KPIs, supplier chart, and grid
- Product detail page per item (stock/coverage, financials, supplier & brands)
- Supplier Dashboard (ranking, score, lead time, critical items, payment required)
- Brand Dashboard (Namq, Breehant, Roastery, Tando, Avola, Laundah) with per-brand drill-down
- Analytics page: category/supplier/cost charts, coverage & lead-time distributions, criticality/ABC/XYZ/FSN pies, stock-vs-demand, Top-20 tables, dead stock analysis, AI-style natural-language insights
- Purchase Planning: automatic recommended order quantity/date/priority/stock-out/arrival/payment, PO generation, PO list
- Alert Center: critical stock, order required, coverage below target, missing supplier/price, long lead time, over stock
- Settings page: SharePoint URL, worksheet name, refresh interval (30s/1m/5m/manual), theme, dashboard title, logo URL, Save/Reload/Test Connection — no code changes required
- Mock role switcher (Admin, Inventory Manager, Warehouse, Purchasing, Viewer) gating settings edits and PO creation
- Dark mode with persisted preference
- Microsoft Graph API integration module, ready to go live once you add Azure AD credentials

## Deferred / roadmap (not in this pass)

These were explicitly scoped out to ship the core experience first — the data model and API layer
already anticipate them:

- Real Azure AD (Entra ID) sign-in / SSO and enforced per-role permissions (current roles are a UI-level mock, not an auth boundary)
- Email notifications (SMTP/Graph Mail) and Microsoft Teams webhook alerts
- Full audit log (login/logout, settings changes, export tracking)
- Multi-language runtime switch (English/Arabic) — data model has a `locale` field reserved for this
- Backup & restore UI, multi-company switcher UI (the settings schema already supports a `companies[]` array)
- Movement history is logged via `data/movement-history.json` and an API route, but there's no UI to browse it yet beyond the API

## Getting started

```bash
npm install
npm run dev
```

Open http://localhost:3000 — the app redirects to `/dashboard`. On first data request, it
automatically generates a realistic sample dataset (~110 items across all 6 categories, 6 brands,
10 suppliers) into `data/sample/Master.xlsx` on worksheet `Master File July-26`, so there is
nothing else to configure to try the app.

## Connecting to a real SharePoint Excel file

1. In Azure Portal, register an app under Entra ID → App registrations.
2. Add **Application permissions** (not delegated): `Sites.Read.Write.All` and `Files.ReadWrite.All`
   (Microsoft Graph), then click **Grant admin consent**.
3. Create a client secret under Certificates & secrets.
4. Copy `.env.example` to `.env.local` and fill in:
   ```
   AZURE_TENANT_ID=...
   AZURE_CLIENT_ID=...
   AZURE_CLIENT_SECRET=...
   ```
5. Restart the dev server, go to **Settings** in the app, switch **Data Source Mode** to
   "Live SharePoint (Microsoft Graph)", paste the workbook URL and worksheet name, click
   **Test Connection**, then **Save**.
6. The expected URL shape is:
   `https://<tenant>.sharepoint.com/sites/<site>/Shared Documents/<file>.xlsx`

If a live fetch ever fails (bad URL, missing permission, expired secret), the app automatically
falls back to the sample dataset rather than showing a blank screen, and the Settings page will
show the connection error from Test Connection.

## Excel column mapping

The parser (`src/lib/excelSource.ts`) expects these exact worksheet headers, matching the source
spec: `Item Code, Description, Item Description, Namq, Breehant, Roastery, Tando, Avola, Laundah,
Used by # Brands, Monthly Demand, Current Stocks, Coverage, Order Status, Criticality, Minimum
Stock Level, Maximum Stock Level, Order QTY, Lead Time, Classification, Price, Supplier, Payment
To Reach Optimum, Monthly Stock Cost`. Brand columns accept `Yes/No`, `Y/N`, `True/False`, or `1/0`.

Category is derived automatically from the Item Code prefix — no column needed:

| Prefix | Category |
|---|---|
| CO | Consumables |
| PM | Packaging Materials |
| RM | Raw Materials |
| FA | Fixed Assets |
| GB | Green Beans |
| FPB | Finished Products |

## Architecture notes

- **Frontend**: Next.js 14 App Router, TypeScript, Tailwind CSS, AG Grid Community, ECharts (via `echarts-for-react`)
- **Backend**: Next.js Route Handlers (Node runtime) under `src/app/api/*` — no separate server process needed
- **Data**: `src/lib/excelSource.ts` reads either the local sample workbook or, in live mode, calls `src/lib/graphClient.ts` (MSAL + Microsoft Graph `workbook/worksheets(...)/usedRange`)
- **Analytics**: `src/lib/analytics.ts` computes KPIs, health score, ABC (Pareto 80/95 on annualized consumption value), XYZ (demand-variability heuristic), FSN, dead stock, alerts, purchase recommendations, and AI-style insights — all derived at read time so any Excel structure change (as long as headers match) flows straight through with no code changes
- **Settings persistence**: flat JSON files under `data/` (`settings.json`, `purchase-orders.json`, `movement-history.json`) — trivial to swap for a real database later
- **Scale**: the sample dataset is ~110 rows for demo purposes; AG Grid's client-side row model comfortably handles tens of thousands of rows. For 100k+ rows in production, switch AG Grid to its Server-Side Row Model and add pagination to the `/api/inventory` route — the current API shape (items array + meta) is compatible with that change.

## A note on how this was built

This codebase was generated in an environment without a local Node.js runtime available to run
`npm install` / `npm run build` for verification. The code follows standard, well-established
Next.js/TypeScript/AG Grid/ECharts patterns throughout, but it has **not been compiled or run**
before delivery. `next.config.js` sets `typescript.ignoreBuildErrors` and `eslint.ignoreDuringBuilds`
to `true` so a stray type or lint issue won't block your first build — but please run:

```bash
npm install
npx tsc --noEmit
npm run build
```

and fix anything that surfaces (should be minor, if anything — mismatched prop names, an unused
import, etc.) before treating this as production-ready. Once it builds cleanly, flip those two
flags back to `false` in `next.config.js` to keep type/lint safety nets on going forward.

## Deploying

Any Next.js host works (Vercel, Azure App Service, Azure Static Web Apps + Functions, self-hosted
Node). Set the three `AZURE_*` environment variables on the host if you want live SharePoint mode
in production; otherwise the app runs fully on sample data with zero configuration.
